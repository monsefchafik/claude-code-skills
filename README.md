# claude-code-skills

> **Claude forgets everything between sessions. These skills make it remember.**

A collection of skills and commands for [Claude Code](https://claude.ai/code) that give Claude
a structured, persistent memory of your project — and a repeatable method for understanding
codebases it has never seen before.

Drop them into any project and get structured codebase analysis, architecture documentation,
and persistent project memory — out of the box.

---

## The problem this solves

Using Claude Code without structure works — until it doesn't:

- Every developer prompts Claude differently → inconsistent results
- Knowledge discovered in a session disappears when it closes
- No one can tell "in what state is this codebase?" without weeks of manual analysis
- Claude re-reads the same code over and over across sessions

These skills encode methodology into Claude. Same steps, same order, same deliverables — regardless of who runs them.

The architecture documentation system reduces the architecture context loaded per session by **−97%** compared to naively importing full analysis files — while giving Claude a more accurate picture of the codebase than any manual summary would.

## What's included

### Skills

| Skill | What it does | Invoke with |
|-------|-------------|-------------|
| [`codebase-analysis`](skills/codebase-analysis/SKILL.md) | Reverse-engineer business logic, domain model, risks and integration points from an existing codebase. Produces a structured risk register with P0/P1/P2 findings anchored to file:line references. | `/codebase-analysis` |
| [`architecture-document`](skills/architecture-document/SKILL.md) | From an analysis conversation or a markdown file, produce the 3-file documentation set: `full-analysis.md`, `summary.md`, `decision.md`. Wires into `architecture-map.md`. | `/architecture-document` |
| [`architecture-reviewer`](skills/architecture-reviewer/SKILL.md) | Isolated subagent that reads a mechanism's `summary.md` and returns an actionable implementation recommendation — without polluting the main context window. | Via Agent tool |
| [`memory-maintainer`](skills/memory-maintainer/SKILL.md) | Keep project memory concise, consistent and non-contradictory across `CLAUDE.md`, `.claude/rules/`, `.claude/memory-bank/` and `docs/ai/`. Promotes durable learnings from conversations into the right memory layer. | `/memory-maintainer` |

### Commands

| Command | What it does | Invoke with |
|---------|-------------|-------------|
| [`memory-bootstrap`](commands/memory-bootstrap.md) | Initialize the full memory system on an existing project: analyzes the codebase, proposes a file plan, generates `CLAUDE.md`, `docs/ai/`, `.claude/rules/` and `.claude/memory-bank/` — all grounded in actual project content. | `/memory-bootstrap` |
| [`promote-to-memory`](commands/promote-to-memory.md) | Promote durable learnings from the current conversation into the right memory layer. | `/promote-to-memory` |

### About rules

Rules (`.claude/rules/`) are **not distributed as generic templates** in this repo.

They are project-specific artifacts — generated by `memory-bootstrap` from your actual codebase.
Running `/memory-bootstrap` on your project will produce rules adapted to your stack, your patterns,
and your known issues. Two projects will get two different rule sets.

What you get here is the **tooling that generates them**, not a one-size-fits-all rule pack.

## Installation

### Option A — everything at once

```bash
cp -r skills/* .claude/skills/
cp -r commands/* .claude/commands/
```

### Option B — clone and symlink (stays up to date)

```bash
git clone https://github.com/<your-handle>/claude-code-skills ~/claude-code-skills

# In your project
ln -s ~/claude-code-skills/skills .claude/skills
ln -s ~/claude-code-skills/commands .claude/commands
```

> **Prerequisite:** Claude Code installed. Skills live in `.claude/skills/<skill-name>/SKILL.md`
> and commands in `.claude/commands/<command-name>.md` inside your project.

## Quick start

### Step 1 — First time on any project — initialize the tools

```
/init
/memory-bootstrap
```

`/init` must run first — it sets up the `.claude/` structure that all skills depend on.
Then `/memory-bootstrap` analyzes your repo, proposes a memory file plan, asks for confirmation,
and generates `CLAUDE.md`, `docs/ai/`, `.claude/rules/` and `.claude/memory-bank/` — all populated
with real content from your codebase. Rules are generated for your stack, not copied from a generic template.

Run this once per project, whether the codebase is new or existing.

### Step 2 — Analyze a codebase

```
/codebase-analysis
```

Claude will start by mapping the repo into units with risk levels, then adapt its approach to the size of the codebase:

| Size | Criteria | Approach |
|------|----------|----------|
| **Small** | < 5k lines, ≤ 3 units | All phases in one pass, single conversation |
| **Medium** | 5k–30k lines, 3–8 units | Phases by layer, single conversation |
| **Large** | > 30k lines, > 8 units | One conversation per HIGH/MEDIUM unit |

For each unit analyzed, Claude will:
1. Ask you to confirm or adjust the scope before reading any code
2. Run phases: domain model → business logic → integrations → risks
3. Produce a risk register with P0/P1/P2 findings, each anchored to a file:line
4. Enter a review loop where you can deep-dive any finding
5. Offer to persist the analysis as architecture documentation

### Step 3 — Persist the analysis as structured architecture docs

```
/architecture-document
```

Converts the analysis conversation into the 3-file documentation set — `full-analysis.md`,
`summary.md`, `decision.md` — and wires everything into `architecture-map.md`.
From that point on, Claude reads `summary.md` before touching the documented mechanism.
Also surfaces the PROMOTE-CANDIDATEs identified during analysis, ready for Step 4.

### Step 4 — architecture-reviewer fires automatically from here

Once Step 3 is done and `architecture-map.md` is wired into `CLAUDE.md`, the
`architecture-reviewer` activates transparently during any normal Claude Code session
that touches a documented mechanism.

Claude reads `architecture-map.md` at session start, recognizes the mechanism, and
lazy-loads `summary.md` via an isolated subagent — before proposing anything.
No manual invocation needed.

### Step 5 — Promote findings into permanent memory

```
/promote-to-memory
```

Dispatches every PROMOTE-CANDIDATE into the right memory layer — rules, patterns, known issues,
architectural decisions. Run it after each significant session.

For periodic cleanup and drift detection across all memory files:

```
/memory-maintainer
```

Audits the full memory system, detects contradictions and stale content, and proposes
reviewable patches before writing anything. Run every 2–4 weeks.

## How everything relates

### Tool ecosystem

```
/codebase-analysis      →   /architecture-document   →   /promote-to-memory
(explore & understand)       (capture in memory)           (promote to rules)
        │                            │                              │
  analysis-plan-*.md       docs/ai/architecture/            .claude/rules/
  (session tracking)       full-analysis.md                 systemPatterns.md
        │                  summary.md                        known-issues.md
        │                  decision.md                       architecture-decisions.md
        │                  (## PROMOTE-CANDIDATEs)                  │
        │                  architecture-map.md          ← wired into CLAUDE.md
        │                            │
        │                  architecture-reviewer
        │                  (reads summary.md before any modification)
        │
        └── /codebase-analysis doc=<file>   ← resume across sessions
```

### When to use what — project lifecycle cadence

```
First time on the project (once)
  └─ /init  →  /memory-bootstrap
       → generates CLAUDE.md + docs/ai/ + .claude/rules/ + .claude/memory-bank/

After each codebase-analysis session
  └─ /architecture-document   → persists the 3-file doc set + wires architecture-map.md
  └─ /promote-to-memory doc=<analysis-file>
       → dispatches PROMOTE-CANDIDATEs into the right memory layer

During any normal Claude Code session touching a documented mechanism
  └─ architecture-reviewer fires automatically (wired via @architecture-map.md in CLAUDE.md)
       → lazy-loads summary.md in an isolated subagent before proposing anything
       → no manual invocation needed — happens transparently

After each significant session
  └─ /promote-to-memory       → promotes decisions and patterns from the conversation

After each merged PR or milestone
  └─ /memory-maintainer (apply mode)
       → rotates activeContext entry to progress.md

Every 2–4 weeks
  └─ /memory-maintainer (audit → patch → apply)
       → catches staleness, detects drift, promotes durable observations
```

## Knowledge that survives session close

A risk register is a report. What makes these skills different is what happens *after* the report.

Both `codebase-analysis` and `architecture-document` produce **PROMOTE-CANDIDATEs** — findings
explicitly tagged for promotion into your project's permanent memory, each pre-routed to exactly
where it belongs:

```
- "Always sort by ID before saveAll() — causes ORA-00060 deadlock"
  → .claude/rules/database-operations.md        [high — prod incident, grep-confirmed]

- "XxxEntity.fromDomainModel() / toDomainModel() pattern across 8 entities"
  → .claude/memory-bank/systemPatterns.md        [high — grep-confirmed in 8 distinct files]

- "PricingEngine API failure aborts pipeline — no fallback"
  → docs/ai/known-issues.md                      [high — Phase 4 failure mode]
```

Candidates are stored in the persisted analysis file — not just in the conversation.
Close the session. Come back in three days. Run `/promote-to-memory doc=<analysis-file>`
and every tagged finding lands exactly where it belongs.

**Nothing waits to be forgotten.**

### The full loop — from finding to permanent memory

`/promote-to-memory` and `memory-maintainer` take it from there.
They read the PROMOTE-CANDIDATEs and dispatch each one into the right layer of your project's
memory hierarchy — rules that constrain every future session, patterns Claude follows when
writing new code, known issues it checks before touching fragile areas, architectural decisions
it never second-guesses:

```
PROMOTE-CANDIDATE found during analysis
    │
    ├── invariant (repo-wide)     → .claude/rules/              ← active at every session
    ├── recurring pattern (2+)    → .claude/memory-bank/systemPatterns.md
    ├── operational risk          → docs/ai/known-issues.md
    ├── architectural decision    → docs/ai/architecture-decisions.md
    └── integration constraint    → docs/ai/integration-patterns.md
```

### Claude Code as a team member

This is what the system is actually building toward.

Imagine a senior developer who was present for every incident, every architectural decision,
every refactoring. Who remembers why you chose one approach over another, what caused the
deadlock in production, which patterns your team has validated and which ones you've explicitly
rejected. A developer who reads the relevant context before every task — and never suggests
something you've already tried.

That's what a fully loaded memory system turns Claude Code into.

Not a code autocomplete. A team member that brings the right context to every task — architecture
evolution, bug investigation, refactoring, QA — without you having to re-explain the project
from scratch each time.

## Stack compatibility

The **methodology is fully stack-agnostic** — phases, domain model extraction, risk register,
PROMOTE-CANDIDATEs, memory layers. None of this depends on a specific language.

The **grep patterns** used during analysis phases are tuned for Java / Spring Boot. On other
stacks, a handful of search patterns may need minor adaptation to match your framework's
conventions (e.g. replacing `@Service` / `@Repository` with their equivalents). The methodology
itself requires no changes — only the search terms.

| | Java / Spring Boot | Other stacks |
|--|-------------------|--------------|
| Methodology | ✅ out of the box | ✅ out of the box |
| Grep patterns | ✅ production-tuned | ⚠️ minor adaptation needed |
| `memory-bootstrap` stack detection | ✅ `pom.xml`, Maven, Oracle | ✅ `package.json`, `requirements.txt`, etc. |

Verified on: Java 21, Spring Boot 3.x, Oracle, Maven.

## Going deeper

Each skill ships with reference files that serve distinct purposes. Reading them in the right
order makes the difference between "it works" and "I understand why it works."

---

### Reference files — inside each skill and command

Every skill and command has a `references/` folder with up to three files:

#### `user-manual.md`
The entry point for new users. It answers: *"When should I use this skill, and how do I invoke it?"*
It contains a routing table (use / don't use), invocation examples with all arguments, and
common usage patterns. Read this before `SKILL.md` — it is shorter and tells you if the skill
is the right tool for your situation.

#### `report-examples.md`
Concrete output examples for each phase of the skill. Useful for two things:
- **Before adopting**: understand what you will actually get — cartography output, phase deliverables,
  risk registers, end-to-end flow synthesis — before running anything
- **When debugging**: if your output looks different, compare against the examples to spot where
  the analysis diverged

#### `dependencies.md`
The contract between tools. For each skill, it documents:
- what files it reads and under what conditions (blocking vs. non-blocking)
- what files it writes and when (automatic vs. user-confirmed)
- what it explicitly does NOT do (the non-write contract)
- compatibility checks to run before modifying an adjacent skill

Essential reading before modifying a skill or wiring a new tool into the system. It is also
the document that explains why `codebase-analysis` never writes to `architecture-map.md` directly
— and why that separation matters.

Commands have the same structure. `memory-bootstrap/references/` contains a `user-manual.md`
(the 7-step workflow at a glance, routing table) and a `dependencies.md` that is particularly
important: it documents the hard dependencies on `memory-maintainer`'s templates — the three
`memory-bank` files must use the exact structure defined in those templates, otherwise
`memory-maintainer` will not recognize the format it is supposed to maintain. A change in
`memory-maintainer`'s templates silently breaks bootstrap output — `dependencies.md` is the
explicit contract that prevents this.

`promote-to-memory/references/` documents the two promotion channels (conversation vs. `doc=`
file), the exact format expected for `PROMOTE-CANDIDATE` items, and when not to promote (simple
Q&A, speculative observations, content already present). The routing table in `user-manual.md`
answers the most common adoption question: *"Should I run this after every session?"*
(Answer: only after sessions that produced decisions, confirmed patterns, or discovered issues —
not after exploratory conversations.)

### Pattern documentation — the "why" behind the system

Three documents explain the design intent of the system. They are not required reading to use
the skills, but they are required reading to adopt the system at team scale or to contribute.

#### [`REVERSE_ENGINEERING_PATTERN_EN.md`](docs/REVERSE_ENGINEERING_PATTERN_EN.md)
The design rationale behind `codebase-analysis`. Explains why the output gate exists (analysis
and documentation are separate acts), why one conversation per unit is a feature not a limitation,
why the analysis plan is a file and not a conversation artifact, and why `🚫 ignored` is a
persistent architectural decision. Also documents the full PROMOTE-CANDIDATE routing table with
confidence levels, and the complete workflow by repo size (small / medium / large / iterative).

Read this when: you want to understand why the skill behaves the way it does, or when you're
considering changing it.

#### [`ARCHITECTURE_DOCUMENTATION_PATTERN_EN.md`](docs/ARCHITECTURE_DOCUMENTATION_PATTERN_EN.md)
The prerequisite wiring guide for `architecture-document` and `architecture-reviewer`.
Documents the exact block to add to `CLAUDE.md` to make the system active — without it,
`architecture-map.md` is never consulted and `architecture-reviewer` is never invoked naturally.
Explains why the instruction must be imperative ("read this BEFORE proposing anything") rather
than suggestive, and why `architecture-reviewer` must always run as an isolated subagent via
the Agent tool.

Read this when: setting up the system on a new project, or when documented mechanisms are not
being consulted despite being in `architecture-map.md`.

#### [`MEMORY_ARCHITECTURE_EN.md`](docs/MEMORY_ARCHITECTURE_EN.md)
The authoritative reference for the 4-layer memory architecture. Documents what belongs in each
layer, the decay rate of each file, the responsibility matrix (what bootstrap generates vs. what
memory-maintainer maintains vs. what only a human can provide), silent drift patterns, and the
healthy maintenance cadence. Includes the complete file tree contract that bootstrap and
memory-maintainer both enforce.

Read this when: onboarding a team to the memory system, debugging why Claude ignores a rule or
repeats a past mistake, or when memory files have grown stale and you need to triage.
